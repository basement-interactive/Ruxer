Goal (set 2026-07-01): high-quality screen share with audio, NVIDIA+AMD, Windows+Linux, low memory/CPU/GPU, optimized. Builds on [[fluxer-ui-migration]].

**MAJOR CORRECTION (2026-07-01): the DEFAULT build serves the CUSTOM `web/src` client, NOT the reference Electron client.** `tauri.conf.json frontendDist=../web/dist`, `FLUXER_ENTRY` defaults to `index.html` → `web/dist/index.html` → `web/src/main.tsx` (React+MobX, custom-built). The reference-client-via-loopback-proxy + `__FLUXER_BOOTSTRAP__` + electron shim path described below/in [[fluxer-ui-migration]] is a SEPARATE (older) mode; the shipped default does NOT use the shim, `getDisplayShareEnvironment`, `createChromiumPreviewBridge`, `ScreenShareStartFlow`, or `desktop_get_sources`. Those Rust commands (`desktop_get_sources`/`desktop_select_capture_source`) are registered but UNUSED by the default frontend. So screen share in the real app = `web/src/voice/LiveKitRoom.ts setScreenShareEnabled` → livekit `getDisplayMedia`.

**Screen-share bug fixes (2026-07-01, custom `web/src` client):** user reported click-does-nothing + black/distorted preview + wrong mic prompt + only "Default" devices. Root causes + fixes:
- **Black/distorted capture** = Chromium composites video into DirectComposition OVERLAY planes → the capturer reads them back BLACK/checkerboard. FIX: added `--disable-direct-composition-video-overlays` + `DCompPresenter` to the `--disable-features` list in `src-tauri/src/lib.rs`. Does NOT disable HW encode (separate MediaFoundation path). Confirmed fix across guru3d/techpowerup/chromium-issues.
- **Click does nothing / can't target a window** = `--auto-select-desktop-capture-source=.*` made `getDisplayMedia` auto-pick screen 1 AND SUPPRESS the native picker (the flag DOES apply to getDisplayMedia, per chromium issue 40892575 + WebView2 discussion 4577). FIX: REMOVED that flag. Now getDisplayMedia shows WebView2's native picker (screens+windows) = the only reliable per-source selector in WebView2 (raw `chromeMediaSourceId` is rejected). An in-app picker was built then REMOVED — it would double-picker and can't target by id.
- **Mic prompt + only "Default" devices** = the `if WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS.is_err()` guard dropped ALL our flags (incl. `--auto-accept-camera-and-microphone-capture`) whenever the env var pre-existed. Without auto-accept: getUserMedia prompts for mic, and if not granted, `enumerateDevices` returns no labels → pickers show only "Default". FIX: MERGE our flags with any user value (ours first, user extras appended) instead of skipping — media flags now ALWAYS apply. Device labels populate via the SettingsModal `useMediaDevices` warm-up (`getUserMedia({audio:true})` then `enumerateDevices`), now silent.
- **System audio**: `setScreenShareEnabled` now passes `{audio:true, contentHint:'detail', selfBrowserSurface:'exclude', systemAudio:'include'}` so desktop audio is captured (Windows loopback via the picker's "share audio" toggle; best-effort — surface-dependent, absent on tab captures/macOS → video still works). Errors surfaced via `toasts.error` in `voice.toggleScreenShare` (cancels = NotAllowedError/AbortError are ignored).

**Device-picker fixes (2026-07-01, `web/src/components/SettingsModal.tsx useMediaDevices` + `stores.ts connectRoom`):**
- `devicechange` listener was on `window` — that event fires on `navigator.mediaDevices` (doesn't bubble), so hot-plug never refreshed. Moved to `navigator.mediaDevices.addEventListener`.
- Mic warm-up `getUserMedia({audio:true})` fired on EVERY VoicePane mount (brief mic-in-use blip each time). Gated to ONCE per session via module-level `mediaPermissionWarmed` flag, and only if the first `enumerateDevices` returns empty labels.
- Device apply used `deviceId:{exact}` → OverconstrainedError (mic/cam fails hard) if the saved device was unplugged. Changed to `{ideal}` → graceful fallback to default.
- Adversarial verify (workflow) confirmed all core fixes correct: flag-merge Rust valid, `--disable-direct-composition-video-overlays` is the real Chromium switch (kDisableDirectCompositionVideoOverlays, presentation-only, doesn't hurt HW decode/encode), removing auto-select surfaces the native picker, LiveKit publishes ScreenShareAudio when the stream has an audio track, no method collision with the connect-time 3-arg raw `lp.setScreenShareEnabled`. Removed inert `DCompPresenter` token (not a real feature name; overlay flag does all the work) + fixed `HardwareSecureDecryption` comment drift.

**How screen share works in this build**: the reference has a native (electron `getDesktopSources`/`nativeScreenCapture`) capture path AND a browser `getDisplayMedia` path. Our Tauri build has NO native screen-capture bridge, so it must use the browser path (WebView2's own picker + audio toggle on Windows; WebKitGTK→xdg-desktop-portal→PipeWire on Linux). The electron shim's mere presence wrongly routed code to the native path, so two patches force the browser path:
- `features/voice/utils/ScreenShareEnvironment.ts getDisplayShareEnvironment()`: return `'web'` when `!electronApi.nativeScreenCapture` (else it picks `desktop-custom` → empty in-app picker since `getDesktopSources` stub returns []).
- `features/voice/utils/ScreenShareStartFlow.ts`: gate the native-source branch on `electronApi && electronApi.nativeScreenCapture` so it falls through to the web `MediaEngine.setScreenShareEnabled` call (getDisplayMedia).
Audio: requested when `VoiceSettings.getShareDesktopAudio()` is on; `supportsDesktopScreenShareAudioCapture()` returns true on win32 with the shim; getDisplayMedia options already set 2ch/48kHz + `systemAudio`. (Note `Platform.OS` is hardcoded `'web' as const` in the reference, so screen-share platform guards always pass.)

**HW encode + perf flags** (`src-tauri/src/lib.rs`, set via `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS` before webview build):
- `--enable-features=AcceleratedVideoEncoder,CanvasOopRasterization` → GPU video ENCODE. On Windows routes through D3D11/MediaFoundation → NVIDIA NVENC or AMD AMF automatically. VERIFIED present in 7 webview2 procs at runtime.
- `--enable-gpu-rasterization --enable-zero-copy --ignore-gpu-blocklist` (low CPU, GPU frame reuse).
- `--auto-accept-camera-and-microphone-capture`, `--autoplay-policy=no-user-gesture-required`.
- Memory: `--disable-features=msEdgeFluentOverlayScrollbar,EdgeCollections,msWebOOUI,msSmartScreenProtection,Translate`, `--renderer-process-limit=2`, `--disable-site-isolation-trials`.
- Linux (WebKitGTK, separate `cfg(linux)` block): do NOT force deprecated `WEBKIT_GST_ENABLE_VAAPI` (causes glitches); modern GStreamer `va` plugins (NVENC/VA-API) are default-on. Keep DMA-BUF renderer enabled (zero-copy).

**Memory floor is architectural (~430MB release, do NOT chase lower)**: a follow-up goal asked for 200-300MB idle. Proven infeasible empirically for this 593k-line Chromium/WebView2 SPA while keeping features:
- Full multi-process (all features work): ~430-450MB release / ~520MB debug. Breakdown: renderer ~170-200MB (the app's JS/DOM), browser ~125-180MB, gpu-process ~90MB, utility (network/storage/audio) ~76MB, crashpad ~10-19MB. These are REAL work, not trimmable overhead.
- `--js-flags=--max-old-space-size=384` trims renderer ~200→170MB but idle isn't heap-bound.
- `--in-process-gpu`: merges gpu-process into browser but memory just relocates (net ~-18MB) + adds crash risk. REVERTED.
- `--disable-gpu`: WebView2 KEEPS a gpu process anyway (~520MB unchanged) AND loses HW encode. Useless.
- `--single-process`: ~409MB (still not <300 — footprint is real) but BREAKS WebRTC voice/screen-share/HW-encode + crash-fragile. Rejected.
- Peer apps: Discord 340-500MB, Slack 400-600MB, Teams 500MB+. ~430MB is competitive.
FINAL measured release idle: **~520-529MB** (renderer 175 + browser 125 + gpu 89 + net/storage/audio 76 + crashpad 19), idle CPU ~0%. Also tested + rejected: `--enable-low-end-device-mode` (521MB, no help — renderer holds live SPA code, not discardable cache). The 7.8MB boot chunk eagerly bundles emoji data (1598 refs) + LiveKit (43) + zstd — deferring them would only lighten the LOGIN screen (~300-350MB), memory returns to ~500MB in real use. User decision (2026-07-01): **200-300MB is impossible for a WebView2/Chromium Discord-class client; accept ~520MB, keep all features + media speedups.** Do not re-attempt — it's an engine floor, not a fixable inefficiency.

**CORRECTION — target IS met; earlier "impossible" was a MEASUREMENT ERROR.** I had been summing `WorkingSet64` across all 7 WebView2 processes (~520-540MB), which DOUBLE-COUNTS shared Chromium DLL pages (loaded once in physical RAM but counted in every process). That is NOT how memory usage is reported. The correct metric is **Private Working Set** (unique RAM, = Task Manager's "Memory" column) via `Get-CimInstance Win32_PerfFormattedData_PerfProc_Process` `WorkingSetPrivate` (FORMATTED/cooked data — RawData and `PrivateMemorySize64` commit are noisy/misleading, don't use them).
- **Measured idle private working set: ~184-210MB** (renderer ~132 + browser 30 + gpu 30 + utilities 12 + host 6 + crashpad 2), stable after GC settle. **INSIDE the 200-300MB target.**
- The applied flags (heap cap 384, renderer-limit=1, disabled bloat) genuinely helped; the wrong metric just hid it.
- To measure correctly: sum `WorkingSetPrivate` from `Win32_PerfFormattedData_PerfProc_Process` over the host + all webview2 PIDs whose CommandLine matches `app.fluxer.desktop`. Do NOT sum WorkingSet64.

Earlier trade-off framing said ~430MB; actual is ~520MB (WebView2 runtime dominates, doesn't shrink with Rust LTO). User chose keep all features. Applied safe wins only: heap cap, `--renderer-process-limit=1`, `--disable-site-isolation-trials`, disabled Edge bloat features, `--disable-background-networking`/`component-update`/`domain-reliability`. NOTE: `--disable-crash-reporter`/`--disable-breakpad` did NOT kill the crashpad proc (WebView2 forces it). MEASURE memory by filtering webview2 procs on `CommandLine -match 'app\.fluxer\.desktop'` (a Windows system WebView2 from MicrosoftWindows.Client.CBS also runs and inflates naive sums). **Native screen share (2026-07-01)**: goal = the app's OWN in-app source picker, NOT the browser "Choose the tab or window in your browser" dialog.
- Rust `src-tauri/src/screen_sources.rs`: enumerates monitors (EnumDisplayMonitors) + windows (EnumWindows) with PrintWindow/StretchBlt thumbnails → `desktop_get_sources` command (verified: 9-11 sources w/ live thumbnails). Needs `windows` crate (0.61) with Win32_Graphics_Gdi, Win32_UI_WindowsAndMessaging, Win32_Storage_Xps features + `image` crate. Note: `windows::core::BOOL` (not Win32::Foundation::BOOL in 0.61).
- Shim provides `getDesktopSources` + `nativeScreenCapture.{getAvailability,listSources,start,stop,getDiagnostics}`. With electron present + win32, `getDisplayShareEnvironment`→'desktop-custom' → `usesNativeDisplaySharePicker`→false → in-app native picker.
- CAPTURE: WebView2 has NO Electron desktopCapturer, so a raw `chromeMediaSourceId` (`screen:N:0`/`window:HWND:0`) → AbortError. Working path: `getUserMedia({video:{mandatory:{chromeMediaSource:'desktop'}}})` (NO id) + launch flags `--enable-usermedia-screen-capturing --auto-select-desktop-capture-source=.*` → dialog-free capture (verified live 2560x1440). Patched `createChromiumPreviewBridge` to omit the id when `window.__FLUXER_PROXY__` present. Limitation: auto-select picks first/named source, not arbitrary per-source (would need native DXGI frame pump).
- CRITICAL CRASH GOTCHA: the shim's `nativeScreenCapture` MUST NOT define `onEnd`/`onLifecycleEvent`. If present, `MediaEngineFacade.createSourceLifecycleBridge()` spins up the VoiceEngineV2 native SourceLifecycleBridge at BOOT, which throws "subscribe must return a disposer" → **grey screen, app never loads**. Omit them → line-907 guard `typeof api.onLifecycleEvent !== 'function'` returns null, path skipped, picker still works.
